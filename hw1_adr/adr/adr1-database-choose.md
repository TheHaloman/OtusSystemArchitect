## Описание
ADR-001: выбор базы данных
## Статус
Принято
## Контекст
Система должна хранить данные о поездках, которые агрегируются из внешних источников (авиа, отели, авто) с различной структурой данных. При этом у нас есть жесткие связи User -> Trip -> Reservation. Нам также нужно удалять поездки с дашборда по завершении поездки и поддерживать международный масштаб (10k+ активных пользователей с потенциалом роста)
## Возможные решения
Чистая Реляционная БД (PostgreSQL)
Документоориентированная БД (MongoDB)
Гибридный подход: SQL для профилей и связей + NoSQL для деталей самих карточек бронирования
## Решение
Выбираем PostgreSQL как основную базу данных
Модель «User -> Trip -> Reservation» идеально ложится на реляционную структуру. Потеря связи для пользователя критична
Для хранения специфических данных от разных поставщиков (авиакомпаний, отелей) будем использовать тип JSONB. Это даст гибкость NoSQL внутри надежной SQL-базы
нужно фильтровать поездки по датам («завершенные») и группировать их. SQL справляется с этим эффективнее и предсказуемее
Синхронизация с внешними системами начисления баллов требует гарантии того, что данные либо записаны полностью, либо нет (ACID)
## Последствия
Плюсы:
- надежность критических связей за счет реляционной модели
- гибкость форматов данных разных вендоров за счет JSONB
- целостность за счет ACID
- эффективные инструменты партицирования по датам поездок
- высокая экспертиза на рынке

Минусы:
- изменения модели данных = миграция
- записи не такие быстрые, как в NoSQL, потребуются дополнительные ADR для критических сценариев с взрывным потоком событий требующих записи данных
