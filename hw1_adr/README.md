## Ката
Ката: https://nealford.com/katas/kata?id=TheRoadWarrior

A major travel agency wants to build the next generation online trip management dashboard to allow travelers to see all of their existing reservations organized by trip either online of through their mobile device.
- Users: 10,000+ registered users worldwide
- Requirements:
  - The system must interface with the agency’s existing airline, hotel, and car rental interface system to automatically load reservations via frequent flier accounts, hotel point accounts, and car rental rewards accounts.
  - Customers should be able to add existing reservations manually as well.
  - Items in the dashboard should be able to be grouped by trip, and once the trip is complete, the items should automatically be removed from the dashboard.
  - Users should also be able to share their trip information by interfacing with standard social media sites.
  - Richest user interface possible across all deployment platforms
- Additional Context:
  - must integrate seamlessly with existing travel systems
  - partnership deals are being negotiated to create 'favored' vendors
  - must work internationally

## 0. Бизнес контекст
Мы крупное туристическое агентство, предоставляющее широкий перечень услуг бронирования для частных клиентов по всему миру. В настоящий момент востребована платформа единого управления планами путешествий с интеграций сопутствующих услуг бронирования. В нашей компании имеются отдельные решения по бронированиям разных услуг с интеграцией на популярных поставщиков. Мы хотим разработать единую экосистему путешественника, объединяющую существующие функции бронирований с предложениями от популярных продавцов. Наша цель: привлечение и удержание клиента за счет предоставления удобных инструментов управления путешествиями, возможностью делиться через социальные сети и системой лояльности. Экономическая целесообразность обусловлена комиссиями с посредничества, а также партнерскими соглашениями для продвижения предпочтительных поставщиков услуг

## 1. Бизнес-цели и бизнес-драйверы
Бизнес-цели:
1. За год нарастить число зарегистрированных клиентов по всему миру до 30 тысяч пользователей
2. Не менее 100 тысяч переходов из социальных сетей из путешествий, которыми делятся наши пользователи
3. Сокращение времени бронирования всей поездки на 30% по сравнению с бронированиями в отдельных сервисах
4. На 20% больше пользователей наших сервисов совершают полный цикл бронирования услуг путешественника через наш сервис по сравнению с бронированием через отдельные сервисы
Бизнес-драйверы:
1. растущая конкуренция интеграторов туристических услуг
2. растущая сложность поддержки отдельных систем и сбора аналитики
3. растущие затраты на продвижение отдельных решений
4. упущенная выгода, когда клиент приобретает только один вид услуги, а остальные бронирует у других поставщиков

## 3. Пользовательские истории

Пользователь:
1. Я как посетитель хочу ознакомиться с предложениями по бронированию, чтобы выбрать наиболее подходящие
2. Я как посетитель хочу составить план поездки, чтобы ознакомиться с итоговыми условиями
3. Я как посетитель хочу стать пользователем через регистрацию или вход, чтобы оплатить план поездки
4. Я как пользователь хочу просматривать сводку по активным и предстоящим поездкам на главном экране приложения, чтобы быстро получать доступ к информации
5. Я как посетитель хочу делиться своими поездками в социальных сетях, чтобы получить +100 социального одобрения и зависть соседей и коллег

Поставщик услуг:
1. Я как поставщик услуг хочу продавать свои услуги через площадку, чтобы получать прибыль
2. Я как поставщик услуг хочу предоставлять свои предложения, тарифы и акции, чтобы площадка предоставляла клиенту актуальную информацию
3. Я как поставщик услуг хочу, чтобы площадка предоставляла верифицированные данные клиента для соблюдения правил оформления документов
4. Я как поставшик услуг хочу, чтобы площадка предоставляла согласие клиентов на обработку персональных данных

Менеджер:
1. Я как менеджер хочу управлять программами продвижения
2. Я как менеджер хочу управлять тарифами
3. Я как менеджер хочу управлять доступностью услуг и приоритезацией партнеров
4. Я как менеджер хочу получать доступ к аналитике

## 4. Атрибуты качества

| Атрибут качества | Требования |
|---|---|
| Производительность (Performance) | - 95% поисковых запросов должны выполняться за 2 секунды - 99% за 5 секунд |
| Масштабируемость (Scalability) | До 5 тысяч одновременных активных сессий и до 50 тысяч в высокий сезон |
| Доступность (Availability) | - система должна быть доступна 99,7% времени - система должна быть доступна в том числе при недоступности внешних систем бронирования |
| Надежность (Reliability) | - Данные о подтвержденных поездках должны гарантированно храниться и восстанавливаться после сбоев не более чем за 1 час - Данные об оплатах должны гарантированно храниться и восстанавливаться после сбоев не более чем за 1 час |
| Расширяемость (Extensibility) | Добавление нового типа услуги не более чем за 1 месяц Добавление нового поставщика не более чем за 2 недели |
| Тестируемость (Testability) | Покрытие автотестами не менее 70% |
| Развертываемость (Deployability) | - Релиз должен занимать не более 1 часа, ограничений по периодичности релизов нет - Релиз не должен приводить к остановке обслуживания пользователей - Откат релиза не должен приводить к остановке обслуживания пользователей |
| Отказоустойчивость (Fault Tolerance) | - обеспечить доступ к поездкам даже при полном отказе внешних систем бронирования - отказ одной из внешних систем не должен приводить к отказу остальных |

## 6. Критические сценарии и критические характеристики

### Сценарий №1: Взрывная нагрузка

#### Описание
Популярный блогер публикует ссылку на план поездки в канале на 1 млн подписчиков
#### Стимул
Резкий скачок трафика, в течение первых 2 минут на страницу переходит 50 тысяч уникальных пользователей
#### Реакция
Система задействует кэширование на CDN и автомасштабирование инстансов в облаке
#### Метрики
Процент успешных запросов (HTTP 200) > 99.9%, время загрузки страницы < 3 секунд
#### Критические характеристики
1. масштабируемость - система справилась с растущей нагрузкой за счет автобалансировки нод в облаке, процент успешных запросов не упал, среднее время ответа осталось в допустимых пределах
2. эластичность - система спрашивалась как с быстрым ростом нагрузки, так и с последующим уменьшением нагрузки, ресурсы в облаке выделялись в соответствии с динамикой изменения нагрузки, не допустив переиспользования лишних ресурсов с одной стороны и ухудшение среднего времени ответа и процента успешных запросов с другой стороны

![Критический сценарий 1](/assets/hw1_adr/diagrams/взрывная%20нагрузка.png)

### Сценарий №2: Недоступность внешней системы бронирования

#### Описание
Система должна сохранять работоспособность и предоставлять пользователю доступ к имеющимся данным при частичном или полном отказе API внешних партнеров (отелей, авиакомпаний)
#### Стимул
Внешняя система (например, API отеля) перестает отвечать, отдает ошибки 5xx или отвечает дольше установленного тайм-аута (например, > 5 секунд)
#### Реакция
1. Размыкается Circuit Breaker, прекращая запросы к упавшей системе.
2. Система переключается на выдачу кэшированных данных.
3. В интерфейсе отображается уведомление о временной неактуальности данных.
4. Система отслеживает восстановление недоступной системы бронирования периодическим опросом
#### Метрика
1. Автоматическое восстановление связи с внешним API в течение 1 минуты после возобновления его работы.
2. 0% ошибок 500 для конечного пользователя при падении внешних систем.
#### Критические характеристики:
1. Доступность: Доступность страницы дашборда не страдает, пользователь видит старые данные (кэш), интерфейс отвечает за < 2 сек
2. Отказоустойчивость: доля успешных отображений данных в дашборде 100% - используются старые данные
